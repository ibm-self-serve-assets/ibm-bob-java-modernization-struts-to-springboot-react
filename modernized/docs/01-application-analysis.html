Legacy Application Analysis — PKS Auto Insurance

Artifact ID: 01_application_analysis  |  Analysis Date: 2025  |  Analyst: IBM Bob

1. Executive Summary

PKS Auto Insurance is a Struts 1.3 monolithic web application deployed as a WAR file. It manages customer registration, vehicle management, insurance quoting, and administrative operations. The codebase relies on end-of-life frameworks (Struts 1.3, Spring 5.2, Hibernate 3.5) running on Java 1.6, presenting significant security vulnerabilities and modernization debt.

Runtime

Java 1.6 (EOL 2013), deployed as .war on Tomcat

Persistence

HSQLDB 1.8 (in-memory/file), Hibernate 3.5 ORM

Web Tier

Struts 1.3 ActionServlet, Tiles 1.3, JSTL 1.2

IoC/DI

Spring 5.2 (wired via XML config), Spring Security commented out

The application follows a classic MVC pattern using Struts 1.3. The ActionServlet is the front controller, dispatching to Action subclasses based on URL mappings in struts-config.xml. Actions call injected Spring services, which delegate to Hibernate-backed DAOs. All request/response objects are Struts ActionForm subclasses. The view layer uses JSP + Tiles layouts with JSTL expressions.

7. Premium Calculation Algorithm

The risk-based premium formula used in PremiumCalculator:

FactorConditionAdjustment
Base PremiumAll policies$500 base annual
Vehicle Ageyear < currentYear - 10+$100 (older vehicle)
Driver Ageage < 25+$200 (young driver)
Driver Ageage > 65+$150 (senior driver)
Make SurchargeLuxury brands+$300
Model YearCurrent year model-$50 (new car discount)
Total—Sum of all applicable factors

The formula is implemented as a series of if-else conditionals. No external rating engine is used. Risk: Business logic tightly coupled to service layer — no unit tests in legacy code.

11. Target Architecture Summary

The modernized platform decomposes the monolith into 3 Spring Boot 3.x microservices (user-service, vehicle-service, premium-service) fronted by an API Gateway, backed by PostgreSQL, secured with JWT + BCrypt, containerised with Docker, and served by a React 18 SPA. Each service owns its schema (plain SQL scripts), exposes REST APIs, and is independently deployable via a DevSecOps CI/CD pipeline. Admin operations (user search, account removal) are part of user-service under a ROLE_ADMIN guard.
