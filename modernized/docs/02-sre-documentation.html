7. Incident Management

Severity Classification

SeverityImpactResponse TimeResolution Target
SEV-1Full service outage, all users impacted5 minutes30 minutes
SEV-2Partial outage, major feature unavailable15 minutes2 hours
SEV-3Degraded performance, workaround available30 minutes8 hours
SEV-4Minor issue, no user-facing impact4 hoursNext business day

7-Step Incident Runbook Template

  1. Detect: Alert fires (PagerDuty / Slack); on-call acknowledges within SLA.
  2. Triage: Identify affected services via Grafana dashboard; classify SEV.
  3. Communicate: Create incident Slack channel #inc-<date>-<sev>; notify stakeholders.
  4. Investigate: Correlate logs (Kibana), metrics (Grafana), traces (Jaeger).
  5. Mitigate: Apply immediate fix (rollback, scale, circuit-break, feature flag off).
  6. Resolve: Confirm SLO metrics recovered; close incident; notify stakeholders.
  7. Post-Mortem: Blameless RCA within 48h; action items tracked in JIRA.

  - type: Resource

13. Key Operational Runbooks

RB-01: Rollback a Service

kubectl rollout undo deployment/user-service -n pks-insurance
kubectl rollout status deployment/user-service -n pks-insurance

RB-02: Scale a Service Manually

kubectl scale deployment/premium-service --replicas=4 -n pks-insurance

RB-03: View Live Logs

kubectl logs -f deployment/vehicle-service -n pks-insurance --tail=100

RB-04: Force DB Connection Pool Reset

kubectl rollout restart deployment/user-service -n pks-insurance

RB-05: Check SLO Error Budget

# Query Prometheus for 30-day error budget remaining
curl -s 'http://prometheus:9090/api/v1/query' \
  --data-urlencode 'query=1 - (sum(rate(http_requests_total{status=~"5.."}[30d])) / sum(rate(http_requests_total[30d])))'
