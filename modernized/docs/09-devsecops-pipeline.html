1. Pipeline Architecture

The pipeline implements Shift-Left Security — security gates are embedded at every stage from commit to production. No stage can be skipped for production deployments. The pipeline is defined in .github/workflows/ci-cd.yml and triggered on all branch pushes and pull requests.

2. Pipeline Stage Definitions

#StageToolActionFailure Policy
① Source Control GitHub / git-secrets Pre-commit hooks: secret scanning (Gitleaks), lint, formatting check BLOCK — secrets found = block push
② Build & SAST Maven + SpotBugs + SonarQube mvn clean package; SpotBugs find-sec-bugs plugin; SonarQube quality gate BLOCK — Critical/High findings fail build
③ Unit Tests JUnit 5 + JaCoCo mvn test; JaCoCo coverage report published; badge on PR BLOCK — any test failure or coverage <80%
④ SCA — Dependency Check OWASP Dependency-Check + Dependabot Scan all Maven dependencies for known CVEs; generate SBOM (CycloneDX) BLOCK — Critical CVE in dependency
⑤ Container Build Docker Buildx + GitHub Container Registry Multi-stage Dockerfile; non-root user; tag with git SHA and semver BLOCK — build failure
⑥ Image Scan Trivy (Aqua Security) Scan Docker image for OS + library CVEs; check for secrets in layers BLOCK — Critical CVE in image
⑦ Deploy to Staging Kubernetes + ArgoCD Apply Helm chart to staging namespace; Flyway migrations run; health checks verified BLOCK — deployment fails health check
⑧ Integration Tests Maven Failsafe + RestAssured Run full integration test suite against staging environment; WireMock not used here BLOCK — any integration test failure
⑨ DAST Scan OWASP ZAP (Zed Attack Proxy) Active scan against staging API; spider all endpoints; check OWASP Top 10 WARN High / BLOCK Critical findings
⑩ Deploy to Production ArgoCD + GitHub Environments 2-approver gate; blue-green or canary deploy; auto-rollback on health failure BLOCK — awaiting approval or health fail

6. Compliance & Audit Requirements

  • 📋 Pipeline Audit Logging: All pipeline executions logged to S3 with actor, timestamp, stage results, and artifact SHAs for 1-year retention.
  • 👥 Two-Person Integrity: Production deployments require approval from two distinct named approvers (GitHub CODEOWNERS + Environments protection rule).
  • 🔏 Image Signing: All production Docker images are signed with cosign (Sigstore); signature verified before ArgoCD deploys to production.
  • 📦 Software Bill of Materials: CycloneDX SBOM generated at stage ④ and attached to every GitHub Release; published to Dependency-Track for continuous monitoring.
  • 🗃️ Immutable Audit Trail: Git history, pipeline logs, and security scan results stored in write-once S3 bucket with Object Lock (WORM).
  • 🔑 Secrets Management: All credentials managed via HashiCorp Vault; no secrets in environment variables or source code; injected at runtime via Vault Agent sidecar.
