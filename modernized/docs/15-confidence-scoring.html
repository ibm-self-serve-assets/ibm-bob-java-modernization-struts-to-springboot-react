Confidence Scoring & Migration Accuracy — PKS Auto Insurance Modernization

Artifact ID: 15_confidence_scoring  |  Project: PKS Auto Insurance  |  Author: IBM Bob

1. Overview

Confidence scoring provides a quantified, per-artifact measure of how accurately each migrated component matches the behaviour and intent of its legacy equivalent. Scores are computed at three levels: individual artifact, component group, and overall project. The score at each level directly drives review-effort allocation: high-confidence artifacts get spot-checks; low-confidence artifacts get senior engineer review with additional test cases authored before sign-off. The overall project score acts as a go/no-go gate for production cutover — the target is ≥ 85.

Scores are not static. They are recomputed after each review phase as issues are found and resolved. The delta between the baseline score (post-generation) and the final score (post-acceptance) measures the value delivered by the human review process layered on top of IBM Bob's generation. In this project the score rose from 71 → 87 across five milestones.

2. Confidence Score Model

Each artifact receives a score between 0 and 100 computed as a weighted composite of four sub-dimensions:

DimensionWeightWhat Is Measured
Structural Similarity30%AST-level comparison of logic structure: control-flow paths, method signatures, call graphs. Measures whether the modern code looks like the legacy code at a structural level.
Behavioral Equivalence40%Test pass rate against legacy-derived test oracles: same inputs fed to both the legacy Struts action and the modern REST endpoint; responses compared for semantic equivalence. Highest weight because behaviour is what the business cares about.
Coverage Score15%Percentage of legacy code paths exercised by the generated test suite. Paths not covered by tests are unverified and lower the score proportionally.
Security Compliance15%Absence of known vulnerability patterns: hard-coded credentials, missing input validation, insecure defaults, OWASP Top 10 patterns. Detected by SAST scan and Security Engineer review.
90 – 100High ConfidenceAuto-approved; spot-check only
75 – 89Medium ConfidenceDeveloper review required
60 – 74Low ConfidenceSenior review + additional test cases
< 60Very Low ConfidenceManual rewrite; flagged as risk item

5. How Behavioral Equivalence Is Measured

Behavioral equivalence testing uses a test oracle approach. Legacy Struts action behaviour is captured as a set of input → expected-output pairs derived from code analysis and exploratory testing of the running legacy system. The same input is then submitted to the modernized REST endpoint. The oracle classifies each result into one of three categories:

  • Equivalent: different format, same semantics (e.g. 302 redirect → 200 JSON — the HTTP mechanics differ but the business outcome is identical)
  • Regression: a genuine behavioural change where the modern endpoint produces a different business result for the same input — must be resolved before sign-off
  • Enhancement: improved behaviour in the modern system (e.g. richer error messages, structured error response) — documented and approved by Domain Expert
