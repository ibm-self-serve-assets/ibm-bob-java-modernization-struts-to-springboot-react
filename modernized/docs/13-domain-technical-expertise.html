1. Executive Overview

Modernizing a production auto insurance system demands two distinct, equally critical competencies: deep domain knowledge of auto insurance workflows, regulatory constraints, and business rules, and deep technical expertise across both the legacy Struts 1.3 / Java 1.6 / HSQLDB stack and the modern Spring Boot 3.x / Java 21 / PostgreSQL target. IBM Bob provided AI-augmented reverse engineering, code analysis, and code generation at machine speed — but every critical gate was staffed with human experts who validated correctness against the real insurance business domain. No migration decision was finalized on AI output alone: domain experts signed off on business rules, and senior engineers signed off on all generated code before merge.

The table below maps the three key expertise dimensions — domain, legacy stack, and modern stack — to the team members responsible, ensuring no knowledge gap was left unattended during the 13-week engagement.

3. Domain Expertise Applied

Auto Insurance Business Rules

The premium calculation engine is the most domain-critical component in the system. An incorrect migration here carries direct financial and regulatory risk. IBM Bob extracted the calculation logic from VehicleServiceImpl.calculatePremium() and generated a dedicated PremiumService; the Domain Expert verified every factor and threshold against current underwriting standards before sign-off.

  • Rating Factors: driver age, vehicle manufacture year, vehicle make, accident history count
  • Age bands: under 25 → high-risk surcharge; 25–65 → standard; over 65 → senior adjustment
  • Vehicle age: vehicles older than 10 years receive an additional premium increment
  • Make surcharge: certain vehicle makes carry higher base rates (luxury, sports categories)
  • Accident history: each recorded accident increments the base premium by a configurable multiplier

Policy Types (Coverage Categories)

Bodily Injury

Covers injury to third parties caused by the insured driver. Mandatory in most jurisdictions.

Collision

Covers damage to the insured vehicle resulting from a collision with another vehicle or object.

Medical Payment

Covers medical expenses for the insured and passengers regardless of fault.

Physical Damage

Covers repair or replacement of the insured vehicle for non-collision causes (fire, theft, vandalism).

Customer Identity & Authentication

  • SSN as primary key: the legacy system used Social Security Number as the natural primary key for the CUSTOMER table. Domain Expert confirmed this is an accepted pattern in insurance systems for identity uniqueness.
  • Role-based access: ROLE_USER (policy holders) and ROLE_ADMIN (operations staff). Admins can search and remove any customer record.
  • Password security: legacy system stored passwords as plain-text in CUSTOMER.PASSWD. Security Engineer mandated BCrypt hashing with strength 12 for all passwords in the modernized system.

Vehicle Management

  • Each vehicle is associated with exactly one customer (SSN foreign key)
  • Vehicle record carries: plate_number, mf_year (manufacture year), make, model, type
  • Vehicle types drive coverage eligibility rules (motorcycle exclusions, commercial vehicle add-ons)
  • Manufacture year stored as VARCHAR(4) — a critical data type decision resolved during project (see Section 6)

Admin Operations

  • Search customers by SSN, name, or email
  • Remove customer accounts with cascade delete on associated vehicles and quotes
  • Audit log requirement: admin actions must be traceable to the performing admin user
  • Admin functionality was consolidated into user-service under ROLE_ADMIN guard rather than a separate microservice (validated by Domain Expert and Architect)

5. Legacy Knowledge Preservation

Legacy systems embed three layers of knowledge. The most dangerous is tacit knowledge — facts that exist only in the minds of original developers. IBM Bob's automated extraction surfaces the explicit layer; domain experts are essential to validate and recover the tacit layer before it is lost permanently during decommission.

Tacit KnowledgeIn developer heads: why certain premium thresholds were chosen, why SSN was the PK, why admin was not a separate service. Recovered via Domain Expert workshops.
Explicit KnowledgeIn source code and config: action mappings, service methods, Hibernate mappings, SQL schema, Spring XML. Extracted and documented by IBM Bob analysis artifacts 01–12.
Documented KnowledgeIn this artifact set (01–15): architecture decisions, domain rules, migration rationale, confidence scores, traceability matrices, runbooks. Persists post-delivery.

IBM Bob processed the entire legacy codebase in Phase 0 and produced structured reverse-engineering outputs (Artifact 01: Application Analysis, Artifact 03: DDD Microservices Design) that made explicit knowledge reviewable by the full team, including domain experts who had never read the source code.

7. Expertise Continuity

A modernization project delivers lasting value only if the client team can maintain and evolve the system independently post-delivery. The following knowledge-transfer plan was executed in parallel with the final two sprints:

Documentation Deliverables

  • Artifacts 01–15: full HTML documentation set covering analysis, architecture, tests, pipelines, setup, domain expertise, migration, and confidence scoring
  • README.md at project root with quick-start, port map, and environment variable reference
  • Inline code documentation: all public service and controller methods carry Javadoc
  • OpenAPI 3.0 specs exposed via Springdoc on each service (/swagger-ui.html)

Runbooks

  • Local Setup Runbook (Artifact 10): Mac / Linux / Windows setup, JAVA_HOME, Docker Compose, health check verification
  • Database Migration Runbook (Artifact 06): Flyway baseline, rollback procedure, manual ALTER TABLE steps for emergency fixes
  • CI/CD Runbook (Artifact 09): Jenkins pipeline stages, OCP deployment, environment promotion gates

Team Enablement

  • Pair programming sessions: IBM Bob–generated code walked through line-by-line with the client Lead Developer
  • Domain Expert sign-off sessions recorded as decision log entries (Section 6 above)
  • QA Lead trained on Testcontainers patterns for adding new integration tests as features evolve
  • Security Engineer produced a quarterly CVE re-scan checklist tied to the Spring Boot and JJWT dependency versions
