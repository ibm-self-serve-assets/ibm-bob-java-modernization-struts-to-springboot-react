Validation Methodology — PKS Auto Insurance Modernization

Artifact ID: 12_validation_methodology  |  Project: PKS Auto Insurance  |  Author: IBM Bob

1. Validation Philosophy

Multi-layered. No single test type is sufficient. Validation spans five layers — from static code analysis at the bottom through unit, integration, functional/E2E, and acceptance testing at the top. A failure at any layer blocks promotion to the next.

Continuous. Every sprint closes with a full validation run. No sprint is considered "done" until its quality gates are green. Validation is not a phase-end activity — it is woven into every pull request, every merge, and every nightly build.

Evidence-based. Every quality gate (G1–G5) produces a signed-off artefact: a build log, a coverage report, a SAST XML, or a product owner sign-off ticket. These artefacts are stored in GitLab CI artefacts and linked from JIRA so that audit trails are complete and immutable.

2. Validation Layers

The stack is read bottom-up: each layer depends on the layer beneath it passing cleanly. A failure at any layer causes an immediate stop and a defect is raised before work continues.

L5
Acceptance Testing
Business stakeholder sign-off on all BF-01 to BF-08 functions. Product owner executes exploratory testing against the staging environment and formally signs the acceptance checklist.
L4
Functional / End-to-End Testing
Full user journey tests driven through the React SPA via a real browser (Playwright). Covers registration → login → vehicle registration → premium calculation → policy view flows.
L3
Integration Testing
Service + PostgreSQL + mocked external dependencies. Spring Boot test slices with Testcontainers. Every IT-* test case from Artifact 08 is executed. DB state assertions confirm data integrity after each operation.
L2
Unit Testing
Isolated business logic tests with Mockito-mocked dependencies. JUnit 5. One test class per service/domain class. Coverage enforced ≥80 % per service at the line level.
L1
Static Analysis
Automated code quality (SpotBugs, PMD, Checkstyle) and security scanning (OWASP Dependency-Check, SonarQube SAST). Run on every push. Zero critical/high findings required to proceed.

Each legacy business function (BF-01 to BF-08) identified during reverse engineering is mapped to one or more modern test cases. A function is considered equivalently verified only when its modern test passes AND a product owner has confirmed the behaviour matches the legacy system's documented specification.

4. Quality Gates

Five sequential gates must each be green before the next phase of work begins. Gate failures are non-negotiable blockers — no exceptions without Change Advisory Board approval and documented risk acceptance.

G1 — Code Compilation
Clean Build
Zero compile errors across all three services and the React frontend. mvn clean package -q and npm run build both exit 0.
G2 — Static Analysis
SAST Threshold
Zero Critical or High SAST findings. Fewer than 5 Medium findings. OWASP Dependency-Check score 0 Critical CVEs in runtime scope.
G3 — Unit Test
Coverage & Pass Rate
100 % of unit tests pass. Line coverage ≥80 % per service (≥90 % for user-service, ≥95 % for premium-service). JaCoCo enforces limits at build time.
G4 — Integration Test
Full IT Suite
All IT-* test cases pass. DB state assertions verified via Testcontainers. No flaky tests tolerated (3 consecutive green runs required).
G5 — Acceptance
Business Sign-off
All BF-01 to BF-08 functions verified and formally signed off by the product owner in JIRA. Each BF ticket transitions to Accepted status. No BF may be deferred past sprint close.

5. Regression Strategy

On every PR merge: Full unit + integration suite (L1–L3) runs automatically in CI. Any regression introduced by the merge is detected before the code reaches main.

Nightly (02:00 UTC): Full regression suite (L1–L4) including Playwright E2E tests runs against the main branch deployed to the staging environment. Nightly failures page the on-call engineer within 5 minutes via PagerDuty.

On every deploy to staging/production: Critical-path smoke test (15 key scenarios covering BF-01 to BF-05) runs before traffic is switched. Smoke failure triggers automatic rollback.

8. Validation Checkpoints Timeline

  • CP-01 — Project Kick-off
    Reverse Engineering Baseline
    Artifacts produced: application inventory, business rule catalogue (draft). Sign-off: Modernization Lead. Gate: none (input to all subsequent gates).
  • CP-02 — Sprint 1 Complete
    Knowledge Model Accepted
    Artifacts: ERD, data dictionary, legacy API catalogue, vulnerability report. Sign-off: DBA + Security Engineer. Gate: G5 (knowledge model review).
  • CP-03 — Sprint 2 Complete
    Domain Model & API Contracts Frozen
    Artifacts: JPA entities, PostgreSQL DDL, OpenAPI 3.0 specs (3 services). Sign-off: Lead Dev + Product Owner. Gate: G1 + G5 (contract review).
  • CP-04 — Sprint 3 Complete
    Microservice Scaffolding + Security
    Artifacts: Spring Boot projects (all 3 services building), JWT filter chain, BCrypt encoder. Sign-off: Lead Dev. Gate: G1 + G2 (SAST on generated code).
  • CP-05 — Sprint 4 Complete
    Business Logic Ported
    Artifacts: Full service-layer implementation (BF-01 → BF-08), unit test suite, JaCoCo report. Sign-off: BA. Gate: G3 (≥80 % coverage all services).
  • CP-06 — Sprint 5 Complete
    Integration Tests Green
    Artifacts: All IT-* tests passing with Testcontainers PostgreSQL, Failsafe XML report. Sign-off: QA Lead. Gate: G4.
  • CP-07 — Sprint 6 Complete
    E2E + Acceptance
    Artifacts: Playwright report (all journeys green), signed JIRA BF-01 → BF-08 tickets. Sign-off: Product Owner. Gate: G4 + G5.
  • CP-08 — Production Go-Live
    Smoke Test + NFR Validation
    Artifacts: Pre-deploy smoke report (15 scenarios green), performance test results, security pen-test sign-off. Sign-off: Product Owner + Security Officer. Gate: All G1–G5 green on production build.

10. Evidence & Audit Trail

Every gate produces an immutable artefact stored in GitLab CI Artifacts with a retention period of 90 days (Critical gate artefacts: 365 days). Artefacts are linked from JIRA tickets so that any stakeholder can trace a deployed feature back through acceptance, integration, unit, and static-analysis evidence.

Gate Evidence Collected Storage Location Retention
G1 Maven/npm build log, exit code GitLab CI Artifacts — build/ 30 days
G2 SonarQube quality gate XML, OWASP HTML report, Dependency-Check JSON GitLab CI Artifacts — sast-reports/; SonarQube project dashboard 365 days
G3 Surefire XML test report, JaCoCo HTML + XML coverage report GitLab CI Artifacts — test-reports/; linked from JIRA story 90 days
G4 Failsafe XML (IT tests), Playwright HTML report, DB diff log, Testcontainers startup log GitLab CI Artifacts — it-reports/ and e2e-reports/ 90 days
G5 Signed JIRA acceptance ticket per BF (PDF export), product owner email confirmation JIRA project — PKS-MODERN board, sprint retrospective report; email archive 365 days

All evidence packages for a release are bundled into a Release Evidence Pack (zip) attached to the JIRA release ticket and archived in the project Confluence space under PKS Auto Insurance / Releases / [version]. The pack is the single source of truth for regulatory or audit inquiries.
