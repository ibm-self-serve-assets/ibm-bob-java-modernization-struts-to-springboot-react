Reverse & Forward Engineering Approach — PKS Auto Insurance Modernization

Artifact ID: 11_reverse_forward_engineering  |  Project: PKS Auto Insurance  |  Author: IBM Bob

1. Overview

Two-phase engineering is the backbone of the PKS Auto Insurance modernization. The goal is to move safely from a Struts 1.3 Java 1.6 monolith to three Spring Boot 3.x microservices without losing a single business rule. The two phases are deliberately separated so that understanding is complete before generation begins.

Phase 1 — Reverse Engineering

Reverse engineering extracts latent knowledge from the legacy codebase — business rules buried in Action classes, data contracts hiding in Hibernate mapping files, security assumptions encoded in XML configuration, and API surfaces scattered across struts-config.xml. The deliverable of this phase is a Knowledge Model: a structured, technology-neutral set of artefacts (catalogues, ERDs, sequence diagrams, vulnerability reports) that faithfully describes what the system does without describing how it does it in Struts terms.

Phase 2 — Forward Engineering

Forward engineering consumes the Knowledge Model and synthesises modern implementation artefacts. IBM Bob generates JPA entities, OpenAPI 3.0 contracts, Spring Security 6 JWT configuration, Flyway migration scripts, JUnit 5 test suites, and Docker/Jenkins CI-CD manifests. Every generated artefact is traceable back to a specific reverse-engineering output, ensuring no knowledge is invented and none is lost.

The two phases are connected by a traceability chain (see Section 4) that provides an audit-ready line of evidence from every line of legacy code to its modern equivalent.

5. Knowledge Preservation

Not all knowledge from the legacy system is carried forward unchanged. Some knowledge is preserved as-is, some is transformed to fit modern idioms, and some is intentionally discarded because it represents implementation detail that does not belong in the modern architecture.

Knowledge Type Legacy Form Modern Form Disposition
Business rules Embedded in Action classes (execute() methods) and service beans Explicit service-layer methods in Java 21 with Javadoc Transformed
Data structures Hibernate 3.5 .hbm.xml POJOs + HSQLDB tables JPA @Entity classes + PostgreSQL 15 schema Transformed
Security requirements spring-security.xml form-login, JDBC user details Spring Security 6 JWT stateless filter chain, BCrypt Transformed
UI workflows JSP pages + Struts ActionForms + tiles layout React 18 SPA components + React Router routes Transformed
Configuration XML-heavy (struts-config.xml, applicationContext.xml) application.yml + Spring Boot auto-configuration Transformed
Test coverage Ad-hoc JUnit 3/4 tests, no coverage enforcement JUnit 5 with ≥80 % line coverage gate enforced in CI Transformed
Struts action chaining Multi-hop <forward> declarations between actions — Discarded (replaced by REST calls)
Session-based state Server-side HttpSession for authentication state — Discarded (stateless JWT replaces)
HSQLDB-specific SQL In-memory HSQLDB DDL dialect, proprietary functions — Discarded (rewritten for PostgreSQL)

7. Risk Management in RE/FE

Risk Phase Mitigation Residual Risk Level
Business logic misinterpretation — AI or analyst misreads intent of a legacy method RE Two-pass review: IBM Bob extraction + human BA sign-off on every BF entry; unit tests derived from BF catalogue Low
Data type mismatches — HSQLDB types differ from PostgreSQL equivalents (e.g. BIGINT vs NUMERIC) RE + FE Explicit type-mapping table produced during RE-03; DBA reviews every column in PostgreSQL DDL Low
Untranslatable Struts patterns — action chaining, tiles inheritance have no direct Spring Boot equivalent RE Patterns are identified in RE-06 and explicitly mapped to REST + React Router equivalents; no automatic translation attempted Medium
Missing test coverage — legacy code has no tests; FE tests may miss edge cases FE Business rule catalogue drives test case generation; coverage gate ≥80 % enforced in CI; product owner acceptance tests supplement automated suite Medium
Security regression — removing session-based auth may introduce JWT misconfiguration FE SAST scan (G2) run on generated security code; token expiry, signature validation, and CORS unit tested explicitly Low
Scope creep during FE — generated code adds unrequested features FE Every FE artefact reviewed against RE input; IBM Bob output diff reviewed before commit; feature freeze enforced post-FE-06 Low
