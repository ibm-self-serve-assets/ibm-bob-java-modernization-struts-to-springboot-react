DDD Microservices Identification & Recommendation — PKS Auto Insurance

Artifact ID: 03_ddd_microservices  |  Pattern: Domain-Driven Design  |  Author: IBM Bob

1. Domain Discovery & Strategic Design

Domain-Driven Design (DDD) was applied to identify natural business boundaries within the PKS Auto Insurance monolith. The domain model was analysed using Event Storming, yielding three domain types: Core, Supporting, and Generic.

Domain TypeDomainRationaleInvestment
Core DomainPremium CalculationPrimary competitive differentiator — unique risk algorithmBuild in-house, highest investment
Core DomainInsurance PolicyVehicle registration and quote binding drives revenueBuild in-house
SupportingIdentity ManagementCustomer auth & profile — important but not differentiatingBuild, consider off-shelf later
SupportingAdministrationInternal tooling for ops teamLightweight build
GenericAPI Gateway / RoutingCross-cutting infrastructure concernUse OSS (Spring Cloud Gateway)

2. Identified Bounded Contexts

BC-1 · Identity Context

user-service

Owns everything about a Customer: registration, authentication, profile management, role assignment, and credential lifecycle.

Aggregates: Customer, Credential, Role

BC-2 · Vehicle Context

vehicle-service

Owns the Vehicle aggregate — make, model, year, VIN — and its association to a customer reference (by ID only, no shared entity).

Aggregates: Vehicle, VehicleOwnership

BC-3 · Premium Context

premium-service

Owns the Quote aggregate and the premium calculation algorithm. Calls vehicle-service and user-service for data; never stores customer state.

Aggregates: Quote, RiskFactor

BC-5 · Gateway Context

api-gateway

Generic routing, JWT validation, rate limiting, CORS, and request/response transformation. Does not own any business domain state.

Pattern: Open Host Service (OHS) — exposes unified API surface to frontend

5. Data Ownership Rules

Each service is the single source of truth for its own data. Cross-service data needs are satisfied by REST API calls, never by shared database tables or cross-schema joins.

8. Decomposition Rationale

PrincipleHow Applied
Single ResponsibilityEach service owns exactly one bounded context and one schema. No service handles two business domains.
Bounded Context AlignmentService boundaries match the DDD bounded contexts discovered during Event Storming.
Independent DeployabilityNo compile-time dependencies between services. Deploy any service without affecting others.
Database per ServiceEach service owns its PostgreSQL schema. No cross-schema SQL joins. Schema migrations are per-service Flyway scripts.
API-First DesignOpenAPI 3.0 contract defined before implementation. Frontend and gateway depend only on the contract.
Strangler Fig PatternMonolith routes are migrated one by one to microservices behind the gateway; legacy app runs in parallel until full cutover.
