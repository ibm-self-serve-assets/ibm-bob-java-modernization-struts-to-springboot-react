1. Migration Strategy Overview

The PKS Auto Insurance modernization used the Strangler Fig pattern as its core migration strategy. Under this pattern, a gateway layer routes traffic to both the legacy Struts monolith and the new Spring Boot microservices simultaneously during the transition period. Capabilities are cut over feature-by-feature; the legacy path is retired only after the modern path has been verified under production-equivalent load. The monolith is decommissioned only when all routes have been validated and the gateway routes 100% of traffic to modern services.

3. Code Conversion Approach

Every component processed by IBM Bob followed a consistent four-step conversion cycle. No generated artifact was promoted to the codebase without passing all four gates.

1
Classify IBM Bob analyses each legacy artifact and assigns a conversion category: auto-convertible, semi-automatic, manual, or excluded.
2
Generate IBM Bob produces the Spring Boot equivalent — controller, service, repository, entity, test, or config — using the classified template.
3
Review Lead Developer and (for domain-sensitive components) Domain Expert review the generated output against the original legacy behaviour.
4
Validate Automated tests (unit + integration) are run. Manual verification against legacy test oracles confirms behavioural equivalence before merge.

The review gate (Step 3) is mandatory and cannot be bypassed, even for high-confidence auto-converted artifacts. IBM Bob's confidence scores (Artifact 15) drive the intensity of review — high-confidence artifacts receive spot-checks; low-confidence artifacts require senior-engineer review with additional test cases authored before sign-off.

7. Data Migration

Schema Transformation

Legacy TableModern TableServiceKey Changes
CUSTOMERusersuser-serviceColumn rename: PASSWD → password_hash (BCrypt); EMAIL → email; SSN retained as PK (VARCHAR(11)); added created_at timestamp
VEHICLEvehiclesvehicle-serviceAdded surrogate id BIGSERIAL PK; MF_YEAR CHAR(4) → mf_year VARCHAR(4) (ISS-003); FK customer_ssn references users.ssn
USER_ROLEuser_rolesuser-serviceJunction table retained; role values standardised to ROLE_USER, ROLE_ADMIN

Password Migration

Legacy CUSTOMER.PASSWD stored passwords as plain-text. BCrypt is a one-way hash — there is no algorithmic path from a BCrypt hash back to a plain-text password, and therefore no in-place migration is possible. The chosen strategy: admin account is seeded with a BCrypt hash during database initialisation; all regular users re-register on first login to the modernized system. This eliminates all plain-text credential risk immediately upon decommission of the legacy WAR.

SSN Handling

In the legacy system, SSN was stored as-is in CUSTOMER.SSN (VARCHAR) and used as the Hibernate entity primary key. The modern system retains this design (validated by Domain Expert — see Artifact 13 §6). The users table uses ssn VARCHAR(11) PRIMARY KEY; SSN is also embedded in the JWT subject claim to identify the authenticated user across services.

Data Validation Checks

  • Column type audit: all HSQLDB CHAR columns reviewed for PostgreSQL bpchar padding behaviour
  • mf_year CHAR(4) changed to VARCHAR(4) in V1__create_vehicles.sql and retroactively via ALTER TABLE (ISS-003)
  • Numeric precision: DECIMAL columns verified for PostgreSQL precision semantics
  • Case sensitivity: PostgreSQL identifiers are case-folded to lowercase; all legacy uppercase column names migrated to lowercase equivalents
